# Performance baseline — 2026-07-13

**Machine-specific numbers.** These are a *before* snapshot for relative
before/after comparison during a perf-optimization effort — NOT absolute
throughput claims. Re-run on the same machine to compare. Windows 11, conda env
`graph-reachability-zanzibar-index`. Graph measured with `paranoia=False` (the
harness default for measurement).

Reproduce:

```bash
PY="C:/Users/avery/anaconda3/envs/graph-reachability-zanzibar-index/python.exe"
# micro-benchmarks (default sizes: depth=8 chains=200 pop=100000 blocked=1000 checks=10000)
"$PY" benchmarks/set_engine_bench.py
# scaling harness over real fixtures (appends JSON records)
"$PY" benchmarks/scale_bench.py --json
```

## Micro-benchmarks (`set_engine_bench.py`)

### Default sizes (depth=8, chains=200, pop=100000, blocked=1000, checks=10000)

| workload | impl | ops/sec | elapsed (s) | peak RSS (MB) |
|---|---|--:|--:|--:|
| (a) deep check | py (PySets) | 8,993 | 1.11 | 64 |
| (a) deep check | roaring | 7,811 | 1.28 | 69 |
| (a) deep check | graph | 286 | 34.9 | 70 |
| (a-write) deep write | py | 392 | 4.60 | 76 |
| (a-write) deep write | roaring | 400 | 4.50 | 76 |
| (a-write) deep write | graph | 38 | 46.9 | 76 |

Second run (noise sample), py + graph: deep check py 9,286 / graph 388;
deep write py 508 / graph 41. Expect ±10% run-to-run.

**(b) wide/flat expand-heavy at pop=100000 did NOT complete** within a 4-minute
cap on this machine (bulk `expand` over a 100k population is the slow path) —
captured instead at a smaller scale below. Worth a dedicated re-measure if
`expand` is an optimization target.

### Small scale (depth=8, chains=100, pop=8000, blocked=1000, checks=3000) — captures (b)/(c)

| workload | impl | ops/sec | elapsed (s) | peak RSS (MB) |
|---|---|--:|--:|--:|
| (a) deep check | py | 17,766 | 0.17 | 63 |
| (a) deep check | graph | 455 | 6.60 | 64 |
| (a-write) deep write | py | 618 | 1.46 | 67 |
| (a-write) deep write | graph | 56 | 16.0 | 68 |
| (b) wide expand | py | 250 | 0.12 | 84 |
| (c) batch membership (reused expand memo) | py | 1,299,545 | 0.002 | 93 |

Headline shape: the set engine reads ~25-30× faster than the graph on the
check-heavy deep workload and writes ~10× faster (the graph pays closure
materialization at write time — the intended cost-model trade). PySets edges out
RoaringSets on these small/medium check workloads.

## Scaling harness (`scale_bench.py`) — appended to `scale_bench.jsonl`

Partial sweep captured this session (the graph configs are very slow to build —
hundreds of seconds — so only a subset ran). Real-fixture `demorgans` workload:

| impl | scale | tuples | build (s) | writes/s | peak RSS (MB) | checks/s |
|---|--:|--:|--:|--:|--:|--:|
| set:roaring | 500 | 54,250 | 72.5 | 749 | 120 | 11,834 |
| set:roaring | 1500 | 162,750 | 216.0 | 754 | 239 | 14,239 |
| graph | 100 | 4,850 | 329.5 | 15 | 65 | 598 |

The graph build cost dominates at scale (329s for 4,850 tuples vs 72s for the
set engine's 54,250) — the closure-materialization write cost is the headline
scaling number to watch. `answers_sig` matches across backends (cross-backend
agreement holds at scale). See `scale_bench.jsonl` for the raw records and
`SCALING.md` for the prior narrative.

## Notes for the optimizer

- Baseline captured before any perf work. The set engine's `check` and the
  graph's closure-materialization write path are the two hottest, most-measured
  surfaces — start there (both are in the high/moderate test-confidence tiers).
- `expand` at large population and the graph build at scale are the slow spots
  not fully captured here; re-measure explicitly if targeting them.
