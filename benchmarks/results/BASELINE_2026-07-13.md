throughput claims. Re-run on the same machine to compare. Windows 11, conda env
`graph-reachability-zanzibar-index`, SQLite in-memory for **both** backends,

## The three schema tiers

- **simple** — a direct-only floor: `type doc { define viewer: [user] }`. No
  hierarchy, no booleans, no stars. Isolates each surface's *constant* per-op cost
  from traversal. 4 tuples/doc.
- **gdrive** — `gdrive.fga`: pure-union folders/groups/docs with a `viewer from
  parent` TTU chain (bounded ancestor depth 5) and `group#member` fan-in. Query
  mix hits `can_read` via owner / direct viewer / group / ancestor folder / ghost.
- **demorgans** — `demorgans_law_2.fga`: a 5-level boolean+TTU derived cascade
  with `and`/`but not` and `[user:*]` stars —
  `access ← authorized_user ← role_user_met ← user_met_requirement ← missing_user`
  over conditions and attributes. Graph maintains derived state via the delta
  processor (`backfill()` at build); the set engine evaluates the booleans
  pointwise.

## Reading it — per surface

**write.** The graph pays its memoization up front: **15–156 writes/s** (closure
materialization + a full `backfill()` reconcile of every derived key on boolean
schemas) vs the set engine's **~780–810 writes/s** — 5–50× slower to load, worst on
the boolean cascade. This is the memoization spectrum working as designed.

**check.** Both are **flat in N** — the headline OpenFGA answer holds. Set-engine
check holds ~78–93k/s (simple), ~16–20k/s (gdrive), ~13–15k/s (demorgans) across
the whole range; a `check` walks only the neighborhood the query touches, not the
store. Graph check is also flat (~390–680/s) but ~25–50× slower in absolute terms
here: every closure probe is a Python→SQLAlchemy→SQLite round-trip, while the set
engine answers from in-process roaring bitmaps with no I/O boundary.

**lookup_reverse.** Also **flat in N** on both backends (set ~1k–34k/s, graph
~100–530/s). Set-engine reverse is `expand` rendered as a result — bitmap algebra
over the object's local fan-in — so it doesn't grow with total tuples.

**lookup — the outlier, and the optimization target.** Set-engine `lookup` is a
**full-store candidate sweep** (one `check` per interned key — O(stored tuples) per
call; `engine.py:821`, a known deferred TODO), so it **degrades ~linearly with N**:

| tier | lookup at ~5k tup | at ~50–67k | at ~100–163k |
|---|--:|--:|--:|
| simple    | 386 (1k) | 37 (10k) | 3.4 (100k) |
| gdrive    | 14.2 | — | 0.8 (67k) |
| demorgans | 62.4 | 12.1 (54k) | 4.0 (163k) |

Each ~10× in tuples costs ~10× in lookup rate. (demorgans compounds it: its result
set also grows with N — mean lookup result 245 → 1,187 → 3,560 ids — so both the
sweep *and* the payload scale.) By contrast the graph's `lookup` walks the
materialized closure, bounded by the reachable set, so it stays flat — and **the
graph beats the set engine on lookup once N is nontrivial**: at simple/1k graph
already leads 534 vs 386, and the gap widens without bound (set → 3.4 at 100k). On
gdrive the graph leads 42 vs 14 at 4.2k. (demorgans/100 is the exception — set 62 >
graph 7.8 at the smallest scale — because the graph pays per-node DB round-trips
over the heavier boolean closure + residue scan; but the set number is already
falling toward the graph's as N grows.) **`lookup` is the one read surface where
materialization wins — the clearest single target for perf work.**

**memory (set engine).** ~linear in tuples: **≈0.9–1.3 KB per raw tuple** (simple
0.86, demorgans 0.93, gdrive 1.29). A 163k-tuple demorgans store fits in ~241 MB
RSS; extrapolating, ~1 M tuples ≈ 1–1.3 GB. The set engine is RAM-bound (all state
in-process, rebuilt on open); the graph lives in the DB and isn't RAM-capped the
same way (it trades RAM for storage + write cost).

## Supplementary micro-benchmark (`set_engine_bench.py`)

Synthetic deep/narrow (nesting depth 8) and wide/flat (star + exclusion) schemas
at depth=8 chains=100 pop=8000 blocked=1000 checks=3000:

| workload | impl | ops/sec |
|---|---|--:|
| deep check | py | 18,095 |
| deep check | roaring | 14,170 |
| deep check | graph | 558 |
| deep write | py / roaring | 760 / 784 |
| deep write | graph | 68 |
| wide expand | py | 381 |
| wide expand | **roaring** | **12,439** |
| batch membership (reused expand memo) | roaring | 1,665,464 |

Roaring vs plain Python sets barely moves the pointwise `check` (small-set
traversals) but is **~33× faster on bulk `expand`** over a large population — the
one place the bitmap backend earns its keep. (`expand` at pop=100000 does not
complete within a 4-min cap on this machine; re-measure explicitly if targeting
it.)

## Cross-backend agreement

Each row records `answers_sig`, the bit-vector of the first 200 check answers as
hex. At every scale-matched pair — simple/250, gdrive/250, demorgans/100 — the set
engine and graph index signatures are **identical**, i.e. the two backends agree
query-by-query, not merely on aggregate true-counts. So every throughput
comparison above is between two backends computing the *same* answers.

## Paranoia cost (graph write path)

Paranoia mode (`make_wildcard_index(paranoia=True)`, boolean spec §8.1) runs the
full I1–I13 invariant checker + the delta-scoped outbox verifier **inside every
commit**. It is a graph-index feature — **the set engine has no equivalent**, so
this measures the graph only. Because it's a per-*commit* cost, the batched matrix
above (one commit at end of build) barely triggers it; to measure it as a real
writer pays it, these rows commit per write (`--commit-every 1`) on the `simple`
tier (per-write commits are invalid on the boolean `demorgans` schema):

| tuples | write /s (paranoia OFF) | write /s (paranoia ON) | slowdown |
|--:|--:|--:|--:|
| 400   | 143 | 30 | **4.8×** |
| 1,000 | 150 | 15 | **10×** |

Two takeaways:

- **Paranoia is a write-only cost.** check/lookup/reverse are unchanged off vs on
  (check ~650–690/s, lookup ~510–570/s, reverse ~500–550/s in both) — the checker
  fires on commit, not on read.
- **Its overhead grows with store size, not just write count.** The per-write
  slowdown rises 4.8× → 10× from 400 → 1,000 tuples because each commit re-checks
  the whole store (O(store) per commit ⇒ ~O(N²) to load N tuples under per-write
  commits). This is exactly why the scaling matrix runs `paranoia=False`, and why
  paranoia is a prerelease-only safety net rather than a production write path.

Reproduce: `--backend graph --commit-every 1 [--paranoia]`.

## Reproducibility

Re-running two batched graph configs (paranoia OFF, default read params) against
the committed baseline, same session:

| config · surface | baseline | re-run | Δ |
|---|--:|--:|--:|
| gdrive/250 build | 85.4 s | 86.4 s | +1% |
| gdrive/250 check | 387.8 | 378.5 | −2% |
| gdrive/250 lookup | 42.3 | 42.2 | 0% |
| gdrive/250 reverse | 99.1 | 98.2 | −1% |
| simple/250 build | 6.41 s | 6.27 s | −2% |
| simple/250 check | 682 | 636 | −7% |
| simple/250 lookup | 534 | 445 | −17% |
| simple/250 reverse | 530 | 535 | +1% |

`gdrive/250` reproduces within <3% on every surface; `simple/250`'s build/reverse
are stable but its check/lookup (both DB-round-trip-bound at only ~500–680/s, so
cache- and load-sensitive) wobble up to ~17% run-to-run. `answers_sig` was
identical across all re-runs. **Rule of thumb for before/after comparisons: trust
set-engine numbers to ~±10% and low-throughput graph read numbers to ~±15%;
anything larger is a real change.**

## Notes for the optimizer

- **`lookup` (set engine) is the standout target.** It's the only surface that
  degrades with N, it's O(stored tuples) by construction, and the graph index
  already demonstrates the flat alternative. A reverse-index / per-subject
  materialization would close it. (Changing the lookup algorithm touches the Lean
  model — see CLAUDE.md's "Perf work & the Lean model" note; a reverse-index is an
  algorithm change, not a behavior-preserving micro-opt.)
- **Graph write (closure materialization + `backfill()`)** is the other slow path —
  15 writes/s on the boolean cascade dominates any graph-at-scale story.
- check and reverse are already flat and fast on the set engine; low leverage.
- Roaring only helps bulk `expand`; don't expect it to move pointwise check/lookup.
- These are in-RAM SQLite numbers; a production graph on PostgreSQL pays network
  latency per probe (worse constant, same O(1)-in-depth), while surviving restarts
  and being shared across readers — things the in-RAM set engine can't do.
