## 1. demorgans graph curves — the R4-BF unlock (newly buildable)

**R4-BF delivered its headline: the boolean `demorgans` graph now builds at every scale.**
Round 3 could not build it *at all* (backfill-bound past the cap even at scale 100). Here
it builds at 100 / 500 / 1500 (4,850 / 54,250 / 162,750 tuples), and the build is
**linear** in tuples (O(N^1.03)) — not the super-linear curve `simple`/`gdrive` show.

Format: `graph / set (ratio)`, ratio = **graph ÷ set** (so **>1 = graph faster**; for build
time it is time so **<1 = graph faster**). Rates ops/s, build seconds, RSS peak MB.

| tuples | graph build | set rebuild | **check** | **lookup** | reverse | graph peak RSS | set peak RSS |
|--:|--:|--:|--:|--:|--:|--:|--:|
| 4,850 | 2.41 s | 0.46 s | 281.6 / 7,344.9 (**0.038×**) | 28.0 / 45.1 (**0.62×**) | 231.8 / 784.5 (0.30×) | 122 | 83 |
| 54,250 | 29.55 s | 5.95 s | 279.2 / 7,093.8 (**0.039×**) | 6.6 / 10.1 (**0.66×**) | 236.6 / 803.9 (0.29×) | 699 | 292 |
| 162,750 | 89.22 s | 17.78 s | 276.8 / 7,371.9 (**0.038×**) | 2.1 / 3.1 (**0.68×**) | 233.6 / 796.1 (0.29×) | 1,955 | 755 |

Reading it:

- **The set engine beats the graph on *every* surface at *every* measured scale on
  demorgans.** check: set ~26× (graph 0.038×). lookup: set ~1.5× (graph 0.62–0.68×).
  reverse: set ~3.4× (graph 0.29×). Build: set ~4–5× faster (graph is 5.0×/5.0×/5.0× the
  time). Peak RSS: graph holds 1.5–2.6× the set engine's.
- **The graph's reads are flat as designed** (check ~277–282, reverse ~232–237, dead flat
  10³→10⁵), but on a wildcard-free boolean schema that buys nothing: the set engine is both
  flat-and-faster on check/reverse and, on lookup, both backends are payload-bound (result
  set grows with N — mean lookup size 380→5,555) so both slide together and set stays ~1.5×
  ahead.
- **This mirrors round 3's `simple` verdict**: after P1 flattened set lookup, a
  wildcard-free schema (even a boolean one) gives the graph index no read-side reason to
  exist. demorgans has no object wildcards, so — unlike `gdrive` — set lookup does not
  sweep, and the graph's one niche (§3) never opens.

Fitted slopes (log-log, 3 scale points; graph:bulk from `benchmarks.analyze`, set:roaring
computed directly — `analyze.py` raises on set rows because `build_s` is `null`, so its
per-op fits below were computed with the same `linfit`):

| impl | build/rebuild | check | lookup | reverse |
|---|--:|--:|--:|--:|
| graph:bulk demorgans | **O(N^1.03)** (R²=1.00) | O(1) (−0.00) | O(N^0.71) (−0.71, R²=0.98) | O(1) (+0.00) |
| set:roaring demorgans | **O(N^1.04)** (R²=1.00) | O(1) (−0.00) | O(N^0.74) (−0.74, R²=0.98) | O(1) (+0.01) |

Both builds are linear; both lookups are payload-bound at ~O(N^0.7) — the same law round 3
reported for set demorgans lookup. R4-BF did not change any read law; it changed *whether
the graph builds at all*.

## 2. 200 k anchors — both graph builds fit the cap this session

Round 3: "200 k did not build within the cap." **This session both 200 k graph builds
completed under the ~10-min cap** — a real R4-BF/P13 improvement, though the `gdrive` one
went deep into swap. Reads used the lighter flags; ratio = graph ÷ set.

### simple, scale 50000 = 200,000 tuples — build FIT time and RAM

| surface | graph:bulk | set:roaring | ratio (graph÷set) |
|---|--:|--:|--:|
| build / rebuild | **453.76 s** (441 t/s) | 13.69 s (14,606 t/s) | build 33× slower |
| peak RSS | 1,292 MB | 907 MB | 1.42× |
| check | 319.9 | 47,828.0 | **0.0067×** (set ~150×) |
| lookup | 390.7 | 10,257.8 | **0.038×** (set ~26×) |
| reverse | 402.0 | 17,394.6 | 0.023× (set ~43×) |

simple graph 200 k built in **453.8 s (< 600 s cap)**, peak **1.29 GB** (comfortably under
free RAM). Set wins every surface by 26–150× and builds 33× faster — the round-3 `simple`
story, now confirmed at 200 k.

### gdrive, scale 12000 = 201,600 tuples — build FIT the cap but hit the RAM ceiling

| surface | graph:bulk | set:roaring | ratio (graph÷set) |
|---|--:|--:|--:|
| build / rebuild | **390.46 s** (516 t/s) | 22.00 s (9,162 t/s) | build 18× slower |
| peak RSS | **3,512 MB** | 916 MB | 3.83× |
| check | 226.3 | 11,686.6 | 0.019× (set ~52×) |
| **lookup** | **170.4** | **0.16** | **1,066×** (graph wins) |
| reverse | 225.7 | 2,168.9 | 0.104× (set ~9.6×) |

gdrive graph 200 k built in **390.5 s (< cap)** but peaked at **3.51 GB working set** with
only **~2.44 GB free** — i.e. it built through heavy swap and survived (no `MemoryError`,
not cap-killed). This is the "memory wall" round 3 anticipated for gdrive/200 k: it did not
kill the run, but the 3.83× RSS blow-up over the set engine is the cost. On a truly
memory-starved box this row would be the one to fail.

**This is the one surface the graph wins, and it wins enormously:** forward `lookup` on the
object-wildcard `gdrive` schema — **170.4 vs 0.16 /s ≈ 1,066×** at 200 k. Set lookup sweeps
O(store) on object wildcards (P1 keeps it, by design) and has collapsed to 0.16/s; graph
lookup stays flat ~170/s. Round 3 measured ~570× at 100 k; the gap widens to ~1,066× at
200 k exactly as the diverging slopes predict. Everywhere else on gdrive the set engine
still wins (check ~52×, reverse ~9.6×, build 18× faster at 3.8× less peak RAM).

## 5. Key ratios to weigh (verdict deferred)

- **Does graph `check`/`lookup` beat set on demorgans at any measured scale? No — not at
  any scale.** check: graph is **0.038×** set (set ~26×) flat across 4.8 k→163 k. lookup:
  graph is **0.62–0.68×** set (both payload-bound). reverse: **0.29×**. Build: **5× slower**
  at **1.5–2.6× the peak RAM**. On this wildcard-free boolean schema the set engine wins
  outright. R4-BF's value here is that the graph *builds at all*, not that it competes on
  reads.
- **The graph's single win persists and widens: `gdrive` forward lookup ≈ 1,066× at 200 k**
  (170.4 vs 0.16 /s), up from ~570× at 100 k in round 3. This is the object-wildcard niche.
- **Both 200 k graph builds fit the cap** (simple 453.8 s / gdrive 390.5 s), where round 3
  could not build either. **simple fit RAM (peak 1.29 GB); gdrive hit the ceiling (peak
  3.51 GB vs ~2.44 GB free) and built only through swap** — the memory wall did not kill it
  this session but is clearly the gdrive-at-scale binding constraint.
- **demorgans graph build is linear (O(N^1.03))**, unlike simple's round-3 O(N^1.66) /
  gdrive's O(N^1.27) — a cheaper build shape, but still 5× the set engine's rebuild.

**Verdict (reviewed 2026-07-15).** The M2 question is now answered in full: **the graph
index has no second read-side win.** Boolean (`demorgans`) checks were the last candidate
surface, and the set engine beats the graph there on every surface at every scale — the
graph's flat-read property is real but never competitive on a wildcard-free schema,
boolean or not. The deployment rule from round 3 stands, now confirmed to 200 k and on
boolean schemas: **use the set engine everywhere except lookup-heavy, object-wildcard,
large-N workloads**, where the graph's flat forward `lookup` is worth it (~1,066× at
200 k and widening; `ConnectedStore.lookup` already routes there). R4-BF's contribution
is *feasibility*, not competitiveness: boolean graph builds went from impossible to
linear-in-N, and both 200 k builds now fit the cap. The binding constraint at scale is
now **bulk-build RAM** (gdrive 200 k peaked 3.51 GB, surviving only via swap) — that
promotes "chunk/stream the bulk-build DP" onto the worklist (N18) for anyone needing
>100 k object-wildcard builds. Round-4 priorities are unchanged: write/apply throughput
(N16, N15) next.
